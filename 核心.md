# CogniAudit (思考者): 基于流式算法的非侵入式认知审计系统

### 1. 项目核心愿景 (Project Vision)

在 AI 工具化的时代，人类极易在海量对话中迷失思维主线。CogniAudit 不关注“AI 替你做了什么”，而关注“AI 如何改变了你的认知”。这是一个非侵入式的“思维黑匣子”，旨在实时监测、捕捉并审计人类与 AI 协作过程中的认知演跃瞬间。

---

### 2. 技术架构与核心逻辑 (Technical Logic)

### A. 实时监测层 (The Monitor)

- **语义向量化 (Embedding):** 使用 Gemini Embedding API 将用户输入的原始文本实时映射至高维语义潜空间。
- **偏移检测 (Drift Detection):** 采用 **ADWIN (Adaptive Windowing)** 算法。
    - **特性:** 在 $O(\log N)$ 时间复杂度下，基于统计显著性（Hoeffding 边界）自动识别“认知离群点”。

### B. 异步审计层 (The Micro-Audit)

- **即时标记:** 当检测到语义偏移（Drift）时，系统自动拦截该节点。
- **异步分析:** 立即触发轻量级 LLM (Micro-Audit) 对该节点进行上下文分析，识别用户在这一刹那的“转念”方向（如：话题切换、立场反转、顿悟）。

### C. 复盘与总结层 (Review & Summary)

- **按需审计:** 对话结束后，用户可一键开启“认知复盘”。
- **全量合流:** AuditLLM 读取所有被标记的“偏移点”及对应的微审计结论，重构出一条“认知演化路径图”。
- **终末辨证:** 产出最终审计报告，直观呈现用户在此场对话中发生的本质转变。

### 4. 为什么这个项目具有竞争优势？

1. **算法驱动而非单纯 Prompt:** 使用 2007 年经典的 ADWIN 算法解决 2026 年的语义漂移问题，具备坚实的数学支撑。
2. **非侵入式交互:** 用户正常聊天，审计静默发生，不增加交互负担。
3. **资产化叙事:** 将碎片化的对话转化为可追溯、可沉淀的“认知资产”。

**1. 动态基准线管理 (Dynamic Baseline Management)**
本模块作为系统的“感知层”，负责对原始对话流进行预处理，提取纯净的语义信号。
• **滑动窗口与锚点自动重置 (Sliding Window & Anchor Auto-Reset)**
    ◦ **基准锚定：** 初始锚点 \(\vec{V}_{anchor}\) 由对话开始前 3 条用户消息 Embedding 的均值确定。
    ◦ **滑动监测：** 实时维护一个长度为 \(k=3\) 的滑动窗口，计算 \(\vec{V}_{window}\) 与 \(\vec{V}_{anchor}\) 的余弦距离 \(d_t\)。
    ◦ **自动重置：** 当 ADWIN 第一次判定漂移时，执行 \(\vec{V}_{anchor} \leftarrow \vec{V}_{window}\)，实现认知阶段的自动切分与滚动审计。
• **静默阈值与非侵入式 UI (Silence Threshold & Silent UI)**
    ◦ **噪音过滤：** 设定 10 字符（约 5 个汉字）的准入阈值，自动忽略“嗯”、“好的”等低语义密度的短文本，防止无效跳动。
    ◦ **完全静默：** 监测过程在后台静默执行，UI 界面不显示任何实时告警，确保用户处于自然、无干扰的“心流”对话状态。
• **仅限用户与去平滑策略 (User-Only & No EMA)**
    ◦ **独立变量：** 审计算法仅针对“用户”的消息进行向量化。AI 的回复作为跟随变量不计入基准，以确保捕捉到的是用户真实的主动思维偏离。
    ◦ **无 EMA 延迟：** 拒绝在输入端进行指数加权移动平均（EMA），避免造成信号滞后。将平滑职能交给滑动窗口，将统计判定交给 ADWIN。

**度量指标：** 采用余弦距离$$d = 1 - \frac{\mathbf{A} \cdot \mathbf{B}}{\|\mathbf{A}\| \|\mathbf{B}\|}$$
作为语义偏移的唯一标量来源。
**2. 统计偏移判定层 (Drift Detection Engine)**
本模块作为系统的“判审层”，负责对处理后的信号进行数学裁决。
• **ADWIN 高感度配置**
    ◦ **算法引擎：** 采用经典的 ADWIN (Adaptive Windowing) 算法。
    ◦ **核心参数：** 设定灵敏度 $\delta = 0.02$。
    ◦ **优化逻辑：** 针对短程对话场景，将显著性水平调至 $0.02$（优于标准的 $0.002$），以提升系统在 2-3 个回合内识别观念转折的响应速度。

## 最终规约（Final Spec / Single Source of Truth）

> 本节为全文唯一权威定义。若与前文叙述有冲突，以本节为准。

### 1) 信号流定义 (Signal Pipeline)

- **唯一标量流来源：** 系统监测的唯一动态值 \(d_t\) 为“锚点向量”与“滑动窗口均值向量”的余弦距离。
- **锚点向量** \(\vec{V}_{anchor}\)
  - **初始**：对话开始前 3 条**用户**消息 Embedding 的算术平均。
  - **更新**：仅在 ADWIN 触发漂移判定时，由 \(\vec{V}_{window}\) 接力重置。
- **滑动窗口均值** \(\vec{V}_{window}\)
  - 取最近 \(k=3\) 条**用户**消息向量的算术平均。
- **距离计算：**
  \[
  d_t = 1 - \frac{\vec{V}_{anchor} \cdot \vec{V}_{window}}{\|\vec{V}_{anchor}\| \|\vec{V}_{window}\|}
  \]

### 2) 漂移点索引与重置策略 (Drift Indexing & Reset)

- **ADWIN 输入：** 实时输入上述 \(d_t\)。
- **判定参数：** \(\delta = 0.02\)（高感度模式）。
- **漂移点索引定义：** \(T_{drift}\) 严格定义为“ADWIN **第一次**返回 `drift_detected=True` 时的**当前用户消息索引**”。
- **重置动作：** 当 `drift_detected=True` 时执行 \(\vec{V}_{anchor} \leftarrow \vec{V}_{window}\)。
- **证据窗口范围（Demo 定义）：** \([T_{drift}-2,\, T_{drift}]\) 的用户消息原文（最多 3 条），并附带这些用户消息之后紧随的 AI 回复文本（若存在）。
- **理由：** 使用窗口均值作为新锚点可过滤单次口语波动；将漂移点定义为触发回合可实现“所见即所得”的即时审计体验。

### 2.1) 模型调用与鉴权（Demo 约束）

- **Provider 选择：** Demo 阶段统一使用 **Google AI Studio**（API Key）进行调用，避免 Vertex AI 的 GCP/IAM 配置成本。
- **Embedding 模型：** `text-embedding-004`
- **Micro-Audit 模型：** `gemini-1.5-flash`

## 核心技术规约：触发微审计 (Micro-Audit)

### 1. 输入格式 (Input Protocol)

微审计采用“长短时记忆结合”的接力模式，最大限度降低 Token 消耗并保证逻辑连贯性：

- **主背景 (Primary Context):**
    - 来源于上一次 Micro-Audit 输出的 `Post State`（观念总结）。
    - **冷启动处理：** 若为会话中首次触发，则使用“前 3 条用户原文的总结”作为背景（由 Micro-Audit 先生成一条简短立场摘要）。
- **增量证据 (Incremental Evidence):**
    - 由判定引擎（ADWIN）捕捉到的漂移点证据窗口：\([T_{drift}-2,\, T_{drift}]\) 的用户原文 + 对应的 AI 回复文本。

### 2. 审计任务 (Audit Task)

LLM 审计员的核心指令：

> **分析逻辑：** 对比“主背景”中定义的既定立场，审视“增量证据”中出现的逻辑拐点、新信息摄入或情绪转向，判定两者的偏离程度，并提炼出一段全新的认知描述。
> 

### 3. 输出格式 (Post State JSON)

为了保证后端数据的可追踪性与前端的直观呈现，输出需严格遵守以下精简结构：

**Strict JSON（硬约束）：** 必须仅输出**纯 JSON 字符串**，禁止任何 Markdown/解释性文字/代码块标记。

`{
  "status": "ok | no_meaningful_shift | processing_error",
  "shift_reason": "≤50字：转折诱因（关键冲突/外部启发/情绪转向）",
  "new_perspective": "≤30字：当前最新认知状态（可为空字符串当 status=no_meaningful_shift）",
  "evidence_anchor": ["用户原话1", "用户原话2"]
}`

- 当最新变化不足以形成可复用的新立场（仅语气变化/无新信息）时，返回 `status: "no_meaningful_shift"`，系统应跳过本次 `Audit_History` 入栈。

### 3.1 JSON 自愈策略（实现建议）

- 目标：保证 Demo 期间“心流不断”，避免因偶发格式漂移导致 UI 白屏或路径断裂。
- **阶梯式容错（推荐顺序）：**
  - **正则抢救：** `json.loads` 失败时，先用正则提取最外层 `{...}` 再解析。
  - **单次降温重试：** 若仍失败，立即重试 1 次；重试时将 `temperature=0`，并在提示末尾追加“Reply ONLY with JSON.”一类硬约束。
  - **优雅占位：** 若重试仍失败，返回占位对象并继续 UI 流程，避免中断：
    - `status: "processing_error"`
    - `new_perspective: "检测到深层思维波动，正在解析中..."`
    - `evidence_anchor: []`

### 4. 认知路径的呈现 (Cognitive Path Logic)等用户点击生成认知路径按钮

本系统放弃复杂的拓扑图，采用**“线性演化日志（Evolution Log）”**形式进行序列化呈现：

- **维护机制：** 系统单独维护一个 `Audit_History` 列表，每发生一次漂移，便将新的 `Post State` 压入栈中。
- **直观价值：** 用户点击“查看路径”时，系统按时间顺序排列展示所有的 `Post State`。这种“阶梯式”的输出让用户能直观看到思维是如何从 A 经过“诱因”迁移到 B，再迁移到 C 的完整轨迹。

## 5. 存储架构与隐私设计 (Storage & Privacy)

### 5.1 极简存储方案 (Minimalist Persistence)

为了确保 4.15 Demo 的部署效率与响应速度，系统放弃复杂的云端数据库（如 Supabase），采用 **“会话状态机 (Session State)”** 的完全内存方案：

- **内存实时维护 (In-Memory Management)：**
    - 利用 `Streamlit Session State` 实时维护 `Message_Pool` 与 `Audit_History`。
    - **优势：** 零延迟数据读写，无需复杂的鉴权与网络配置。

### 5.2 隐私与数据安全 (Privacy Boundaries)

- **无痕审计 (No-Trace Policy)：** 默认遵循“随用随弃”原则，所有对话原文与向量数据仅在当前会话内存中存续（无后端数据库、无本地文件落盘）。
- **一键清空：** UI 侧边栏集成“归零（Reset）”功能，点击即瞬间销毁所有内存中的 Embedding 与审计记录，从算法层重置 ADWIN 状态。
